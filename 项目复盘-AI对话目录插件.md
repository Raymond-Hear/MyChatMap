# 我用 AI 做了一个浏览器插件，让 AI 长对话不再难找

> 项目现名：**ChatMap**（原名：AI对话目录插件）

## 起因：一个被反复滚动的痛点

如果你经常用 Kimi、DeepSeek、豆包、腾讯元宝这些 AI 平台，大概有过这样的经历：和 AI 聊了三四十轮，聊到一半想回头找之前讨论过的一个点，结果只能靠记忆拼命往上滚。滚了半天，要么找不到，要么找到了又忘了上下文。

我也经常被这个问题困扰。有一次我在用 DeepSeek 讨论一个技术方案，聊了二十多轮后想回看第五轮关于架构设计的讨论，翻了五分钟才找到。当时就想：如果有个"目录"就好了，像文档的目录一样，点一下就能跳到对应的位置。

于是就有了这个项目 -- **AI对话目录**，一个浏览器插件，给主流 AI 对话平台加上消息级的目录导航。

## 这个项目到底要做什么

说直白一点：在 AI 对话页面的侧边，自动生成一个目录面板。每条用户消息变成一个目录条目（默认取前 8 个字作为标题），点击就能跳转过去，跳过去之后消息还会闪烁高亮一下，让你知道"就是这条"。同时支持重命名目录条目，还能把对话导出成 Markdown 文件保存。

整个项目需要同时适配四个平台：Kimi、DeepSeek、豆包、腾讯元宝。数据全部存在浏览器本地，不上传任何东西。

## 思路是怎么来的

一开始我并没有想清楚怎么做。脑子里只有"给页面加个目录"这个模糊的想法，但具体怎么实现，完全没概念。

**第一个问题：怎么"读懂"页面上的对话？**

AI 平台的页面都是动态渲染的，对话内容是 JavaScript 加载出来的，不像传统网页那样可以直接解析 HTML。我需要一种方式，能自动识别页面上的每一条消息 -- 哪些是用户发的，哪些是 AI 回复的。

最直觉的思路是"找规律"。每个平台的页面结构不一样，但总有一些特征：用户消息通常在右边，AI 回复通常在左边；消息元素有特定的 CSS 类名或者 data 属性。如果我能为每个平台写一套"规则"，告诉插件"看到这种样式的元素就是用户消息，看到那种就是 AI 回复"，问题就解决了。

**第二个问题：四个平台怎么统一？**

Kimi 的页面结构和 DeepSeek 完全不同，豆包又是另一套。如果写四份独立代码，维护起来会很痛苦。所以需要一个统一的"接口" -- 不管底层平台怎么变，上层调用方式是一样的。这就是经典的"适配器模式"：定义一套标准接口，每个平台各写一个适配器来实现它。

**第三个问题：对话内容怎么存？**

对话可能很长，而且用户可能希望下次打开还能看到之前的目录。浏览器的 LocalStorage 容量太小（5-10MB），存不了多少对话。最终选择了 IndexedDB，配合 Dexie.js 这个库来操作，容量几乎无限，而且读写是异步的，不会卡住页面。

## 几个关键的设计决策

**决策一：适配器模式 + 基类继承**

这是整个项目最核心的架构选择。我定义了一个基类 `AIC_BaseAdapter`，里面规定了所有适配器必须实现的接口：`matches()`（判断当前页面是否匹配）、`getMessageList()`（获取所有消息）、`getConversationId()`（获取对话 ID）等等。然后每个平台写一个子类继承它。

比如豆包适配器通过 `data-testid` 属性来识别消息（`send_message` 是用户消息，`receive_message` 是 AI 回复），而 DeepSeek 因为使用了 CSS Modules，类名是一堆乱码一样的 hash 值，只能用部分匹配的方式。腾讯元宝更麻烦，它的 DOM 结构最不稳定，所以适配器里写了四层兜底策略 -- 先查 `data-role` 属性，再查 CSS 类名关键词，再查子元素特征，最后甚至根据布局方向（用户消息通常右对齐）来推断。

这个选择的好处很明显：新增一个平台只需要加一个文件，不用改其他代码。坏处是每个适配器的"读懂页面"逻辑本质上都是和平台 DOM 结构强绑定的，一旦平台改版就可能失效。但这是浏览器插件这类项目的宿命，没有完美解。

**决策二：悬浮球 + 悬停展开的 UI 方案**

PRD 里最初设计的是右侧固定细线，鼠标悬停后展开面板。实际实现时变成了一个可拖拽的悬浮球（FAB），半隐藏在屏幕边缘，鼠标移上去就展开目录面板。

这个方案比固定细线更好用。因为悬浮球可以拖动，用户可以把它放在顺手的位置，不会被页面内容遮挡。拖拽松手后，悬浮球会自动吸附到最近的屏幕边缘。这个交互细节花了不少时间打磨，但效果很自然。

**决策三：用 MutationObserver 监听页面变化**

AI 对话页面的内容是动态的 -- 用户发一条消息，页面上就多出两个元素（用户消息和 AI 回复）。插件需要实时感知这种变化，自动更新目录。

浏览器提供了一个叫 MutationObserver 的 API，可以监视页面上任何元素的增删改。给消息容器装上这个"监控摄像头"之后，只要页面有新消息出现，插件就会在 300 毫秒内（做了防抖处理，避免 AI 打字时频繁触发）自动更新目录。配合 IntersectionObserver 做滚动位置追踪，用户滚动对话时目录会实时高亮当前所在位置。

## 踩过的坑

**坑一：页面加载时机**

插件注入到页面时，对话内容可能还没有渲染完。如果这时候去扫描 DOM，什么消息都找不到。

我的解决方案是"重试三次"：先等 500 毫秒扫描一次，没找到就 1 秒后再扫一次，还没找到就 3 秒后扫第三次。如果三次都失败了，就尝试从 IndexedDB 里加载之前保存的对话数据作为兜底。这个策略虽然不够优雅，但实测下来覆盖了绝大多数场景。

**坑二：SPA 路由切换**

这些 AI 平台都是单页应用（SPA），用户在平台内部切换对话时，URL 会变但页面不会刷新。插件需要感知这种变化，重新初始化。

我用了一个 1 秒间隔的定时器来检测 URL 变化。检测到变化后，会重新选择适配器（因为用户可能从豆包切到了 Kimi），然后重置状态并重新初始化。如果离开了所有支持的聊天页面，还会自动销毁 UI 和事件监听，避免内存泄漏。

**坑三：DeepSeek 的 hash 类名**

DeepSeek 前端用了 CSS Modules，每个类名都是类似 `dad65929`、`fbb737a4` 这样的随机字符串，而且每次版本更新都可能变化。这意味着依赖类名的选择器随时可能失效。

我的做法是只取类名的一部分来匹配（比如只匹配 `fa81`），而不是精确匹配完整的 hash 类名。这样即使类名有轻微变化，也能大概率命中。当然这不是长久之计，理想情况下应该优先使用 `data-testid` 这类稳定的属性，但 DeepSeek 页面并没有提供太多这样的锚点。

## 项目现在的样子

目前项目完成了大约 93%（30 个任务中完成了 28 个），核心功能已经全部可用：

- 豆包和腾讯元宝两个平台的适配器已实现
- 悬浮球 + 目录面板的 UI 交互完整
- 实时目录生成、点击跳转、高亮闪烁
- 条目重命名，支持持久化到 IndexedDB
- 导出当前对话为 Markdown，批量导出为 ZIP
- Popup 面板可以查看已存对话列表
- SPA 路由切换检测

> v0.0.2 版本收缩了支持范围，仅保留豆包和腾讯元宝。Kimi 和 DeepSeek 适配器已移除（DeepSeek 因虚拟滚动导致历史消息抓取不全，Kimi 待后续评估恢复）。

还有 2 个任务待完成：

整个项目的代码量并不大，核心逻辑加起来大约 2000 行 JavaScript，没有使用任何构建工具或框架，纯原生 JS 实现。项目结构也很清晰：

```
background/          # 后台服务，负责数据存取
content/
  adapters/          # 四个平台的适配器
  aic-main.js        # 主入口，核心逻辑
  aic-catalog-ui.js  # 目录 UI 组件
  aic-scroll-tracker.js  # 滚动追踪
  aic-message-bus.js # 消息通信封装
popup/               # 插件弹出面板
shared/              # 公共工具和数据库定义
lib/                 # 第三方库（Dexie.js、JSZip）
```

## 关于 AI 辅助开发

这个项目从 PRD 到代码，全程由 AI 辅助完成。说实话，体验比预想的好很多。

AI 最擅长的是**把模糊的需求变成具体的实现**。比如我说"给 AI 对话页面加个目录"，AI 能直接给出完整的代码架构 -- 适配器模式怎么组织、MutationObserver 怎么用、IndexedDB 怎么设计数据模型。这些东西我自己也能想出来，但可能要花更多时间查资料和试错。

AI 帮了大忙的几个环节：

- **PRD 撰写**：给了基本需求后，AI 产出了完整的产品需求文档，包括用户画像、功能清单、优先级划分、风险分析，结构非常规范。
- **架构设计**：适配器模式、分层设计、消息通信机制，这些架构层面的决策 AI 都能给出合理方案，而且会解释为什么这么做。
- **适配器实现**：每个平台 DOM 解析的具体逻辑，AI 能根据选择器特征快速生成代码，包括多策略兜底。

但也有一些地方 AI 表现不够好：

- **UI 细节打磨**：悬浮球的拖拽交互、吸附动画、面板展开的过渡效果，这些需要反复调试才能达到自然的手感，AI 生成的第一版往往不够好。
- **边界情况处理**：比如页面加载时机、SPA 路由切换、对话切换时的状态重置，这些需要实际测试才能发现的问题，AI 很难提前想到。
- **平台适配的准确性**：AI 给出的 CSS 选择器不一定是最新的，需要对照实际页面验证。DeepSeek 的 hash 类名就改过一次，导致适配器失效。

总的来说，AI 更像是一个能力很强的搭档 -- 你负责想清楚"做什么"和"为什么做"，它负责快速实现"怎么做"。但最终的决策权和质量把控还是得在人的手里。

## 后续计划

v0.0.2 做了一个关键决策：**收缩支持范围，只做豆包和腾讯元宝**。DeepSeek 因为虚拟滚动/懒加载的机制，导致长对话的历史消息无法完整抓取（6 轮对话只抓到 2 轮），这个问题在 DOM 解析层面没有好的解法，暂时搁置。Kimi 也一并移除，先聚焦在两个平台上把体验打磨好。

接下来要做的事：

**优先：补齐剩余功能**，特别是错误提示 UI，让插件完成度达到可以公开发布的状态。

**其次：Edge 兼容性测试**，扩大浏览器覆盖面。

**最后：考虑开源发布到 GitHub**，看看能不能收到其他用户的反馈。等豆包和腾讯元宝稳定后，再考虑恢复 Kimi 和 DeepSeek 的支持（DeepSeek 可能需要走 API 路线而非 DOM 解析）。

做这个项目最大的收获是验证了一个想法：**浏览器插件可以成为改善 AI 平台体验的一种轻量级方式**。不需要平台方配合，不需要 API 权限，只需要读懂页面结构就能提供增值功能。这个思路理论上可以扩展到更多场景 -- 比如给 AI 对话加书签、做对话内容搜索、甚至自动生成对话摘要。但那是另一个故事了。
