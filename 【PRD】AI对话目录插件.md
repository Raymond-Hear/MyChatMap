## 2. 项目背景

### 2.1 问题陈述

在使用豆包、腾讯元宝等国内AI平台时，用户经常进行深度长对话（20轮以上）。当对话变长后，存在以下痛点：

- **定位困难**：需要反复滚动查找历史消息，效率低下
- **结构混乱**：长对话缺乏结构化展示，思路不清晰
- **缺乏备份**：对话内容分散在各平台，难以统一管理和留存

### 2.2 解决方案

开发一款浏览器插件，为多个AI平台提供统一的**消息级目录导航**功能：

- v0.0.2 版本支持豆包和腾讯元宝
- 实时生成对话目录，一键跳转到任意消息
- 纯本地存储，隐私安全
- 支持导出Markdown备份

## 3. 项目目标

### 3.1 核心目标

| 目标 | 衡量指标 |
|-----|---------|
| 提升对话定位效率 | 用户查找历史消息时间缩短50%以上 |
| 跨平台统一体验 | 支持4个主流AI平台 |
| 隐私安全保障 | 0数据上传，100%本地存储 |

### 3.2 MVP范围

**MVP版本包含**：
- 实时自动生成消息级目录
- 右侧固定目录面板，悬停展开
- 点击跳转到对应消息
- 条目重命名功能
- 导出当前对话/所有对话（Markdown/zip）
- 适配Kimi、DeepSeek、豆包、腾讯元宝
- 支持Chrome + Edge

**后续版本**：
- 历史对话处理开关
- 更多浏览器适配（Firefox/Safari）
- 其他增强功能

### 5.2 功能详细说明

#### 5.2.1 目录自动生成

**功能描述**：插件自动识别当前AI对话页面的消息，实时生成目录结构。

**触发时机**：
- 用户发送新消息 → 实时检测 → 自动添加到目录
- 打开已有对话页面 → 检测历史消息 → 生成完整目录

**目录条目命名规则**：
- 默认取用户发送内容的前7-8个字
- 支持用户自定义重命名

**懒加载处理**：
- 检测到页面懒加载时，屏幕中央显示轻提示气泡
- 提示内容："滚动页面加载更多历史消息，以生成完整目录"
- 气泡2-3秒后自动消失

#### 5.2.2 目录UI交互

**位置与样式**：
- 固定在页面右侧边缘
- 默认状态：细线指示条（表示"这里有目录"）
- 悬停状态：展开显示具体目录条目

**条目展示**：
- 条目名：用户第一句话的前7-8个字（或自定义名称）
- 长目录：固定高度区域，支持鼠标滚轮滚动
- 当前位置高亮：用户滚动对话时，目录中对应条目高亮显示

**点击交互**：
- 点击目录条目 → 平滑滚动到对应消息位置
- 跳转后消息高亮闪烁一下

#### 5.2.3 条目重命名

**交互方式**：
- 悬停目录条目时显示编辑图标
- 点击编辑图标 → 就地变成输入框
- 回车确认，ESC取消

**存储**：
- 重命名后的名称存储在本地IndexedDB
- 与消息ID关联
- 下次打开同一对话时显示自定义名称

#### 5.2.4 导出功能

**导出格式（Markdown）**：

```markdown
# 对话标题

**平台**: Kimi / DeepSeek / 豆包 / 腾讯元宝
**时间**: 2024-03-22 14:30
**URL**: https://...

---

## 条目1：用户自定义名称

**用户** (14:30:25):
用户发送的内容...

**AI** (14:30:30):
AI回复的内容...

---

## 条目2：...

**用户** (14:35:10):
...
```

**导出选项**：
- 导出当前对话：单文件Markdown
- 导出所有对话：批量打包成zip

**触发位置**：
- 点击浏览器插件图标 → 弹出面板 → 导出按钮

#### 5.2.5 插件弹出面板

点击浏览器工具栏的插件图标，弹出面板包含：

| 功能 | 说明 |
|-----|------|
| 当前对话目录 | 快速查看/跳转（如果当前页是AI对话） |
| 导出当前对话 | 单文件Markdown |
| 导出所有对话 | 批量zip打包 |
| 历史对话处理开关 | 是否自动处理历史消息（后续版本） |
| 已存对话列表 | 查看/管理本地存储的所有对话 |
| 清理数据 | 删除本地存储的所有数据 |

**面板底部（关于区域）**：
- 插件一句话介绍
- 版本号（灰色小字，如 v1.0.0）
- 作者信息（点击跳转到作者主页）
- 隐私说明："所有数据仅存本地，可放心使用"

#### 5.2.6 错误处理

**页面解析失败时**：
- 显示俏皮短提示（配合插件IP形象）
- 引导用户去GitHub提issue或发邮件反馈
- 示例提示："喵~ 页面好像变了，等我主人更新一下 🐱 [反馈问题]"

## 6. 技术方案

### 6.1 存储方案

**选择**：IndexedDB + Dexie.js

| 对比项 | IndexedDB | LocalStorage | chrome.storage.local |
|-------|-----------|--------------|---------------------|
| 容量 | 数百MB-GB | 5-10MB | 5-10MB |
| 性能 | 异步非阻塞 | 同步阻塞 | 异步 |
| 事务 | 支持 | 不支持 | 不支持 |
| 适用性 | ✅ 大量对话 | ❌ 容量不足 | ❌ 容量不足 |

**数据模型设计**：

```javascript
// 对话表
conversations: {
  id: String,           // 对话ID
  title: String,        // 对话标题
  platform: String,     // 平台标识
  url: String,          // 对话URL
  createdAt: DateTime,  // 创建时间
  updatedAt: DateTime,  // 更新时间
  messages: [           // 消息列表
    {
      id: String,
      type: 'user' | 'ai',
      content: String,
      timestamp: DateTime,
      customTitle: String  // 用户自定义条目名
    }
  ],
  metadata: {
    model: String       // 使用的模型
  }
}
```

### 6.2 适配平台

| 平台 | 匹配URL | 解析策略 |
|-----|--------|---------|
| 豆包 | www.doubao.com | 独立解析逻辑 |
| 腾讯元宝 | yuanbao.tencent.com | 独立解析逻辑 |

> **注意**：Kimi 和 DeepSeek 暂不在 v0.0.2 支持范围内。DeepSeek 因虚拟滚动/懒加载导致历史消息抓取不全，Kimi 待后续版本恢复。

### 6.3 更新机制

- 插件更新时，本地IndexedDB数据自动保留
- 通过Dexie.js版本管理机制处理数据迁移

## 7. 非功能需求

### 7.1 性能要求

| 指标 | 要求 |
|-----|-----|
| 目录生成延迟 | < 500ms |
| 跳转响应时间 | < 100ms |
| 内存占用 | < 50MB |
| 存储 | 无上限，依赖用户磁盘空间 |

### 7.2 兼容性

| 项目 | 要求 |
|-----|-----|
| 浏览器 | Chrome 88+, Edge 88+ |
| AI平台 | 豆包、腾讯元宝（最新版本） |

> **v0.0.2 版本范围**：当前仅支持豆包和腾讯元宝。Kimi 和 DeepSeek 将在后续版本中重新评估支持。

### 7.3 安全性

- 所有数据仅存储在用户本地浏览器
- 不上传任何数据到云端
- 不收集用户隐私信息
- 代码开源，接受社区审计
